A. 개요와 스택

이 코드는 로컬 생성 워크플로우 서버와 연결해 대량 생성, 프롬프트 모듈 관리, 갤러리, 내장 터미널, 로컬 도구 서버를 제공하는 Electron 데스크톱 앱이다. 범위는 개인 생산성 도구보다 넓고, 파일시스템·IPC·로컬 HTTP를 동시에 다루는 특권형 클라이언트에 가깝다.

C. 핵심 근거

1. 코드 품질·유지보수성

  • 🟠 주요 [OVERSIGHT]
    라우팅과 실제 화면 세트가 갈라졌다. src/renderer/src/router/index.ts:4-38와 src/renderer/src/components/layout/AppLayout.vue:44-88는 현재 6개 화면만 노출하지만, DashboardView.vue, BatchView.vue, QueueView.vue, Versions.vue는 참조가 끊겼다. src/renderer/src/views/DashboardView.vue:127-187는 존재하지 않는 /batch로 이동을 시도한다.
  • 🟠 주요 [OVERSIGHT]
    IPC 계약이 중앙 타입에서 이탈했다. src/renderer/src/types/ipc.ts:3-74가 채널 집합을 정의하지만 src/renderer/src/stores/gallery.store.ts:86-92, src/renderer/src/stores/terminal.store.ts:94-129, src/renderer/src/views/QueueView.vue:185-189는 여기에 없는 채널을 직접 호출한다. 타입 시스템이 실제 계약을 완전히 설명하지 못한다.
  • ✅ 강점
    구조 자체는 읽기 쉽다. src/main/services/database/repositories/index.ts:5-32의 필드 화이트리스트, src/main/ipc/validators.ts:1-90의 검증 유틸리티, src/renderer/src/composables/useBatchWizard.ts:1-156의 공통화는 “아무 분리도 없는 바이브 코드”와는 거리가 있다.

2. 보안·신뢰성

  • 🔴 치명적 [OVERSIGHT]
    커스텀 프로토콜이 임의 로컬 파일을 열 수 있다. src/main/index.ts:74-89는 ..만 막고 resolve(filePath)를 그대로 net.fetch(pathToFileURL(...))에 넘긴다. 같은 저장소의 SKILL.md:8-35는 허용 디렉터리 화이트리스트를 요구하지만 실제 구현은 그 단계가 빠져 있다.
  • 🔴 치명적 [OVERSIGHT]
    갤러리 정렬 입력이 SQL에 직접 삽입된다. src/main/ipc/handlers.ts:528-531는 쿼리를 검증 없이 넘기고, src/main/services/database/repositories/index.ts:893-957는 ORDER BY ${sortBy} ${sortOrder}를 그대로 조합한다. 다른 쿼리는 대부분 파라미터 바인딩을 쓰는데, 이 한 곳이 전체 보안 점수를 깎는다.
  • 🟠 주요 [TRADE-OFF: 단일 사용자 로컬 도구]
    로컬 도구 서버는 루프백 한정이지만 권한 경계가 약하다. src/main/services/mcp/index.ts:105-138,171-215는 127.0.0.1에 바인딩하면서도 Access-Control-Allow-Origin: *와 무인증 세션 초기화를 허용한다. 게다가 src/renderer/src/stores/terminal.store.ts:45-53는 첫 터미널 탭 생성 시 서버를 자동 시작하고 설정까지 영구화한다.
  • ✅ 강점
    기초 하드닝은 실제로 들어가 있다. src/main/index.ts:27-35,47-60는 bypassCSP: false, sandbox: true, webSecurity: true를 사용하고, src/main/ipc/validators.ts:1-90와 src/main/services/database/repositories/index.ts:5-32는 입력·필드 화이트리스트를 제공한다.

3. 재현성·운영 준비도

  • 🟠 주요 [OVERSIGHT]
    CI가 현재 기본 브랜치를 지키지 못한다. .github/workflows/ci.yml:3-7는 master만 감시하지만, 실제 작업 브랜치는 main이다. 로컬에선 npm test, npm run lint, npm run build가 통과했어도, 기본 브랜치에서 자동 검증이 빠질 수 있다는 뜻이다.
  • 🟡 경미 [OVERSIGHT]
    메타데이터 드리프트가 보인다. package.json:2-7은 버전 0.15.1인데 package-lock.json:1-10의 루트 버전은 아직 0.11.0이다. 여기에 package.json:6-7, README.md:66-69, electron-builder.yml:48-51의 저장소 URL도 서로 맞지 않는다.
  • ✅ 강점
    로컬 재현성은 강하다. README.md:57-207가 설치·실행·빌드·테스트를 설명하고, package.json:8-25,74-80와 .husky/pre-commit:1는 린트·포맷 훅을 제공한다. 실제 실행에서도 테스트 305건, 린트, 빌드, 감사 스캔이 모두 통과했다.

4. 아키텍처·확장성

  • 🟠 주요 [TRADE-OFF: 단일 사용자 데스크톱]
    현재 구조는 지금 규모엔 맞지만 수평 성장 여유는 적다. src/main/services/database/index.ts:17-81는 메모리 상의 sql.js DB를 통째로 export 해서 저장하고, src/main/index.ts:7-17는 큐, 연결 관리자, 도구 서버를 싱글턴으로 붙인다. 팀형 서비스나 대형 데이터셋으로 확대하면 병목이 선명해진다.

  • ✅ 강점

D. 상위 조치 계획

  1. 하드닝 (Hardening)
    local-asset://를 허용 디렉터리 기반으로 다시 묶어라. src/main/index.ts:74-89를 SKILL.md:24-35 패턴처럼 출력 디렉터리 화이트리스트로 바꾸고, 임의 절대경로·심볼릭 링크·인코딩 우회를 막는 테스트를 추가해야 한다.
  2. 빠른 수습 (Quick Win)
    갤러리 정렬 입력을 IPC 경계에서 화이트리스트 검증하라. sortBy, sortOrder, page, pageSize를 별도 validator로 제한하고, GeneratedImageRepository.list()는 문자열 조합이 아니라 허용된 열 이름 매핑만 사용해야 한다.
  3. 하드닝 (Hardening)
    로컬 도구 서버를 명시적 opt-in 보안 모드로 바꿔라. 자동 시작과 Access-Control-Allow-Origin: *를 제거하고, 적어도 origin allowlist 또는 임시 토큰을 넣어야 한다. 터미널 탭 생성이 서버 상태와 설정을 바꾸는 현재 동작은 분리하는 편이 낫다.
  4. 빠른 수습 (Quick Win)
    CI 브랜치와 메타데이터를 현재 상태에 맞춰 정렬하라. .github/workflows/ci.yml를 main으로 맞추고, package-lock.json 루트 버전과 저장소 URL, 오래된 테스트 수치를 한 번에 정리하면 운영 신뢰도가 즉시 올라간다.
  5. 구조 개선 (Strategic)
    죽은 화면과 문자열 IPC를 함께 정리하라. 더 이상 쓰지 않는 뷰/컴포넌트를 제거하거나 라우트로 복귀시키고, 모든 채널을 src/renderer/src/types/ipc.ts 한 곳에서 강제하도록 바꾸면 유지보수 비용이 크게 줄어든다.
  6. 구조 개선 (Strategic)
    JSON 경계에 스키마 검증을 도입하라. 워크플로우 가져오기, 모듈 import/export, 배치 설정 복원, 도구 서버 입력을 공통 스키마로 검증하고, 지금처럼 catch {}나 빈 객체 반환으로 삼키는 경로를 줄여야 한다.
  7. 구조 개선 (Strategic)
