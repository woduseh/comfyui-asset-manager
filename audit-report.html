🟠 주요: 동적 SQL 필드명 인젝션 벡터 OVERSIGHT
repositories/index.ts — 4개 리포지토리 update 메서드 (라인 115, 229, 294, 361)
Object.keys(data).map(f => `${f} = ?`) 패턴이 필드명을 화이트리스트 없이 SQL에 삽입. 값은 파라미터화되어 있지만 필드명은 미검증. IPC를 통해 렌더러가 임의 필드명을 전달 가능. 실제 익스플로잇 가능성은 낮으나 방어적 코드가 아님.
🟠 주요: 중복 로직 — 배치 위자드 코드 OVERSIGHT
BatchView.vue + JobsView.vue — 모듈 선택, 슬롯 복원, 작업 상태 복구
BatchView와 JobsView가 모듈 매트릭스 추가·슬롯 복원·변수 오버라이드 복원 로직을 각각 ~100줄씩 동일하게 구현. setTimeout(500) 지연 등 매직넘버도 함께 복제됨. 공유 유틸리티 추출이 필요.
🟡 경미: 매직 넘버 산재 OVERSIGHT
main + renderer 전역 — 약 20+개 인스턴스
폴링 간격(3000ms), 청크 크기(50), 재연결 간격(3000~30000ms), 세션 제한(10), 페이지 크기(50), 터미널 폰트(14), 스크롤백(5000) 등이 상수 추출 없이 인라인. 설정 가능해야 할 값들이 하드코딩됨.
🔵 제안: 에러 핸들링 일관성 부족 OVERSIGHT
handlers.ts, App.vue, DashboardView.vue, QueueView.vue 등
일부 핸들러는 상세한 에러 메시지를 반환(ModuleView)하지만, 다른 곳에서는 catch { /* ignore */ }로 조용히 무시. 특히 큐 상태 조회(loadQueueStatus)의 에러 무시는 사용자에게 잘못된 상태를 보여줄 수 있음.

2. 보안 및 안정성 4.5 / 10

🔴 치명적: Electron 샌드박스 비활성화 TRADE-OFF: node-pty 네이티브 모듈 호환성
src/main/index.ts:35-36 — sandbox: false, webSecurity: false
렌더러 프로세스의 격리가 해제됨. sandbox: false는 렌더러에서 Node.js API 접근을 허용하고, webSecurity: false는 동일 출처 정책(SOP)을 비활성화. XSS가 발생하면 RCE(원격 코드 실행)로 직결. node-pty 네이티브 모듈 때문일 가능성 높으나, 이 모듈은 main 프로세스에서만 사용하므로 sandbox 활성화 가능.
🔴 치명적: 로컬 에셋 프로토콜 경로 순회 OVERSIGHT
src/main/index.ts:60-69 — decodeURIComponent 후 경로 검증 없음
local-asset://image/ 프로토콜 핸들러가 디코딩된 경로를 검증 없이 pathToFileURL()로 변환. ..%2F..%2Fetc%2Fpasswd 같은 경로 순회 가능. 또한 bypassCSP: true로 CSP까지 우회하여 공격 표면 확대.
🟠 주요: IPC 핸들러 입력 검증 부재 OVERSIGHT
handlers.ts — 전체 40+ IPC 핸들러
모든 IPC 핸들러가 렌더러로부터 받은 인자를 검증 없이 사용. SETTINGS_SET은 임의 키-값 쓰기 가능, WORKFLOW_UPDATE는 임의 필드 업데이트 가능. Electron에서 렌더러는 신뢰 경계(trust boundary)의 바깥에 있으므로 입력 검증이 필수.
